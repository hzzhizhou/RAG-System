#根目录功能分析
1.config目录用于数据库，redis，评估等的参数配置
2.core目录是项目的主要功能代码
3.log目录用于日志记录
4.until目录是一些工具，来处理数据等

##core目录
1.data_layer 数据层：实现多种文本数据的上传，分块，向量存储，向量数据库初始化
2.retrever_layer 检索层：实现多种检索方式，采用路由策略，查询构建,重排序和HyDE检索增强
3.generation_layer 生成层：提示词模板，生成器构建
4.chat_history_layer 对话记忆层：基于Redis实现长期对话记忆的存储
5.evaluation_layer 评估层：进行RAGAS评估
6.deployment_layer 部署层：提供 API 服务启动能力，具备实际部署和调用价值。

##utils目录
1.text_clean 基础文本清洗：去除控制字符、归一化空白、繁简转换、全角半角转换、移除BOM头等。DocumentLoader 加载后、分块前；QuerySanitizer 中可复用。
2.security 敏感信息脱敏：手机号、身份证、邮箱、密钥等。文档加载后、入库前；生成答案后、日志记录前。
3.metadata_validator 元数据校验与标准化：日期格式（YYYY-MM-DD）、数值范围、枚举值、类型转换。	文档入库前，保证元数据可用于过滤
4.query_sanitizer 用户查询清洗：长度截断（防超限）、去除首尾空格、过滤控制字符、敏感词替换。
5.deduplicator 文档/块级去重：基于内容哈希或 MinHash 相似度，保留唯一副本。 分块后、入库前，减少冗余。
clean_pipeline  将多个清洗步骤串联成管道，便于统一调用。

###核心功能实现

#data_layer,离线阶段向量数据库搭建：加载文档 → 添加基础元数据 → 清洗内容 → 分块 → 入库
1.文档加载，读取
    注意文档类型和大小
    MAX_FILE_SIZE = 10 * 1024 * 1024  # 最大文件大小（10MB，规避内存溢出）
    ALLOWED_EXTENSIONS = [".txt", ".pdf", ".docx", ".xlsx",".md"]  # 允许的文件格式

    添加基础元数据
    载完文档后，立即将基本信息（来源、时间、类型）记录到元数据中，符合“尽早丰富元数据”的原则。后续的分块、清洗、入库等环节都能基于这些元数据做决策

    为什么要用clean_documents函数对文档数据进行清洗？
    1.原始文档常包含大量干扰信息：
    控制字符（如 \x00、\x07）会导致向量嵌入异常或 LLM 输出乱码。
    BOM 头（\ufeff）可能被误判为内容，影响检索匹配。
    全角/半角混用、多余空格会导致相同语义的文本在向量空间中距离变大，降低召回率。
    在读取时立即清洗，确保后续分块、嵌入、检索、生成全流程都基于干净的数据。
    2. 保护隐私与合规
    文档中可能包含 手机号、身份证号、邮箱 等敏感信息。若不在入库前脱敏，则这些信息可能被检索出来并返回给用户，造成数据泄露风险。
    企业需满足 GDPR、等保等法规要求，数据清洗是合规的第一步。
    3. 节省存储与计算成本
    清洗可去除重复内容、无意义字符，减少文档块数量和大小。
    向量数据库存储更小，检索速度更快，LLM 处理时消耗的 Token 更少。
    4.. 提升检索质量与稳定性
    标准化后的文本（如统一繁体→简体、全角→半角）能避免因格式差异导致的语义匹配失败。
    去除干扰字符后，检索器（BM25、向量）能更准确地计算相关性。

2.文档分块
    采用markdown结构化分块和递归分块结合，标明分块的id，方便和原文对比
    注意append方法和extend方法的区别
                final_chunks.extend(sub_chunks)
                final_chunks.append(chunk)
    append：将整个列表作为一个元素添加，适合构建多维结构。（列表中可能有多个元素）
    extend：将列表中的元素逐个添加，用于扁平化合并。

    注意文档分块前后数据类型都是 List[Document]

    例如：docs = [
    Document(page_content="...", metadata={...}),
    Document(page_content="...", metadata={...})
    ]
    只是分块前后的metadata可能不同，分块时可能会添加一些数据。


3.向量入库
    以Chroma为例 Chroma 要求元数据值必须是 str、int、float、bool 或 None。所以Path、datetime 等对象需转换为字符串，入库前需要检查数据类型。

    可以引入内容哈希（如 MD5）来实现增量更新，同时防止同一文档多次入库，浪费资源的现象。


#检索层:实现各种检索方式，实现RAG路由

1.Bm25关键词检索
    首先需要对文档进行分词，这个是关键词检索的核心，因为 BM25 算法依赖词频（term frequency）统计，而中文文本没有自然的分词边界，必须通过分词将连续的字符序列切分成有意义的词语。如果跳过分词，BM25 会把整个文档当成一个“词”，导致检索几乎完全失效（只有查询与整段文本完全一致时才能匹配）。
    这里我用的是utils目录下的tokenizer来实现中文分词。

    然后基于问题给文档片段进行打分排序，输出前top-k个文档（注意：问题也需要进行分词）

    报错分析ValueError: "Bm25Retriever" object has no field "vector_store"出现这样的错误是因为，我的Bm25检索器继承了BaseRetriever,但是BaseRetriever 是 Pydantic 模型，它期望所有子类也遵循 Pydantic 规范。如果你不声明字段，LangChain 的内部机制（如序列化、复制、验证）可能无法正常工作。
    解决办法有：
        一、不继承BaseRetriever类，自己手写实现get_relevant_documents 函数功能。（不推荐，后续无法直接使用 LangChain 的 RetrievalQA 链、EnsembleRetriever 等工具）；
         二、定义 Pydantic 字段。    
            vector_store: Any = Field(description="向量库实例")
            k: int = Field(default=RETRIEVER_K, description="检索返回条数")
            bm25: Any = Field(default=None, exclude=True)  # 不序列化
            docs: List[str] = Field(default=[], exclude=True)
            metadatas: List[dict] = Field(default=[], exclude=True)

3.hybrid混合检索器
    通过导入from langchain.retrievers import EnsembleRetriever这个包，将bm25检索器和向量检索器传入，然后设置两个检索器的权重就实现了。


4.重排序
    主要是对混合检索模式下检索到的相关文档进行重新排序和去重（提高context_precision，上下文精准度）

    这里我采用了分数排序和大模型语义排序两种方式。分数排序顾名思义就是通过对bm25和向量检索相关文档的分数（scores）进行归一化处理，然后按照分数排序就行了；语义排序则需要通过构建提示词，通过大模型来读取这些文档然后按照相关性排序。

5.查询扩展
    引入查询扩展可以提升上下文精准度（context_precision）和回召率（context_recall）
    我写入了两种扩展方式，一是HyDE假设性文档生成检索增强，在检索文档前先利用大模型对用的问题生成一段相关的文段，然后使用检索器进行检索。二是引入了多查询功能，对于复杂的问题，可以将其拆解为多个问题分别回答。

#generation_layer 生成模块
    首先构建生成器的提示词（一份好的提示词可以提高忠实度，减少幻觉；还能提高回答相关性）
    然后实现生成功能，并完成对话记录的存储（这里用到了Redis存储）

#chat_history_layer 对话记录模块
    使用Redis长期会话记忆存储，帮助提供上下文改写功能
    然后实现了记忆读取，添加，删除和查询改写功能。
    最后提供了初始化函数，添加Redis连接失败时启用临时内存存储，
    因为Redis 作为外部中间件，可能因网络抖动、配置错误、资源耗尽等临时故障而不可用。如果此时直接抛出异常，用户的所有请求都会失败，造成服务中断。降级到内存存储后，服务依然能正常处理对话，只是历史记录不会跨会话持久化（重启后丢失）。
